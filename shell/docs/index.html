kiri

A local-first, git-based workflow orchestrator for personal automation.

Drop YAML workflows into workflows/, run kiri in the same directory, and trigger them from a feed-first UI on local.kiri.build.

Install

macOS ARM64 only for now — open an issue if you'd like another platform.

brew install LeeCheneler/kiri/kiri
kiri --version

Homebrew auto-taps LeeCheneler/homebrew-kiri on first install. To upgrade later, brew upgrade kiri.

Without Homebrew

Download the macOS ARM64 binary from the latest release, make it executable, clear the macOS quarantine flag, and put it on your $PATH:

chmod +x ~/Downloads/kiri
xattr -d com.apple.quarantine ~/Downloads/kiri
sudo mv ~/Downloads/kiri /usr/local/bin/kiri
kiri --version

Building from source? See CONTRIBUTING.md.

Quick start

Kiri runs per-directory. Each working directory is its own workspace.

cd ~/projects/some-workspace
kiri init    # scaffold a starter workflow
kiri         # boot on :4242

Then open local.kiri.build and click Run on the starter workflow.

Workflows

Workflows live in workflows/*.yaml. Each has a name and one or more steps:

name: hello
steps:
  - sh: echo "hello world"

Edits are picked up live — no restart needed. The activity feed on local.kiri.build is where runs surface.

Title & grouping — description: / group:

Two optional top-level fields shape how a workflow presents on its page: description: is a one-line summary rendered as the deck beneath the title, and group: is a label (e.g. Dev) shown as the page eyebrow so related workflows read as a set.

name: Patch
description: Patches Dependabot alerts and opens a PR.
group: Dev
steps:
  - sh: echo "patching"

Inline shell — sh:

The simplest step. The string runs under sh -c with a scoped env. Useful for quick scripts that don't warrant a bundle:

name: open-prs
steps:
  - sh: |
      cd "$KIRI_REPO_ROOT"
      gh pr list --state open

Bundle — use:

A bundle is a folder scripts/<name>/ in your repo containing an executable run.sh. A use: step runs that script. Bundles are reusable and version-controlled, and each one defines its own env: contract — the keys it expects (keys starting with KIRI_ are reserved):

name: build
steps:
  - use: build-site
    env:
      TARGET: production

Kiri passes the env: map through verbatim; what each key means is up to the bundle, documented in its own README.md. The project's examples/ directory ships ready-made bundles — including a Claude Code runner — to copy into scripts/ and adapt.

Piping output between steps

Each step's stdout is piped into the next step's stdin. The first step receives empty stdin. A step reads it however it likes — an sh: step with cat, or whatever a bundle's run.sh does:

name: greet
steps:
  - sh: echo "Lee"
  - sh: |
      name=$(cat)
      echo "hello, $name"

Step environment

Every step runs in a fresh per-run scratch directory with a scoped env — nothing from the parent shell is inherited. User env: values apply first; the variables below overwrite on collision, so a workflow can't redirect PATH or shadow kiri's identity vars.

VariableSet onDescription
KIRI_RUN_ID every step The run's ID.
KIRI_STEP_INDEX every step Zero-based index of this step in the run.
KIRI_REPO_ROOT every step Absolute path of the workspace kiri runs against. Steps run in a scratch dir — use this to reach repo files (cd "$KIRI_REPO_ROOT").
KIRI_BUNDLE_DIR use: steps Absolute path to the bundle's scripts/<name>/ directory. Useful for reaching sidecar files shipped with the bundle.
KIRI_RUN_CONTEXT_FILE publish: and summarize: steps Path to a JSON file describing the run so far — workflow name, status, per-step traces, and any published articles.
KIRI_RECOMMENDATIONS_FILE main steps: only Path the step may write JSON Lines to — one proposed follow-up workflow per line. Ingested as recommendations rows on the producing run after the step succeeds. See Recommendations below.
PATH, HOME, USER, LOGNAME every step Passed through from the kiri process so tools that authenticate as you (Keychain, ssh-agent, gpg) keep working.

Keys starting with KIRI_ in your workflow's env: map are rejected at load time — the prefix is reserved.

Workflow inputs — inputs:

An optional list of named parameters collected via a modal when you click Run. One definition can target many things — a single pr-review workflow with a pr_number input reviews any PR, instead of one YAML file per PR. Workflows with no inputs: invoke on a single click as before.

name: pr-review
inputs:
  - name: pr_number
    description: GitHub PR to review
    required: true
  - name: branch
    default: main
steps:
  - sh: echo "pr=$PR_NUMBER branch=$BRANCH"
    env:
      PR_NUMBER:
        input: pr_number
      BRANCH:
        input: branch

Each input is { name, description?, required?, default? }. required: true gates the modal's submit until the field is non-empty; default pre-fills it. Wire an input into any step / publish / summarise env: with { input: <name> } — refs to undeclared inputs fail at load. The resolved input map is snapshotted onto the run, so the activity feed and run page both reflect what the run was invoked with.

Summaries — summarize:

An optional post-run step whose stdout becomes the run's one-or-two-sentence summary, shown on the feed row and at the top of the run page. It has the same { use | sh, env? } shape as a step and runs after steps: and publish:. A failing summariser does not change the run's status.

name: hello
steps:
  - sh: echo "hello world"
summarize:
  sh: echo "said hello to the world"

A summarize: step receives empty stdin and the KIRI_RUN_CONTEXT_FILE environment variable (see the table above), so it can describe what the run did. For an AI-written summary, the examples/ directory ships a claude-code-summarizer bundle — copy it into scripts/ and reference it with use:.

Published articles — publish:

An optional array of long-form markdown articles produced by the run. Each entry has the same { use | sh, env? } shape as a step, plus a name (lowercase letters, digits, hyphens — unique within the workflow) and an optional title. Publish steps run serially after steps: and before summarize:; each step's stdout is captured as the article body. They receive empty stdin — to build an article from the run's data, read KIRI_RUN_CONTEXT_FILE:

name: status
steps:
  - sh: echo "all systems go"
publish:
  - name: report
    title: Daily Status
    sh: |
      echo "# Daily Status"
      echo
      echo "All systems reporting healthy."

Articles surface as chips on the activity feed row, in a Recently Published list in the right-hand column, and under a Published section on the run page. Clicking a chip opens a dedicated /runs/:id/published/:name page that renders the markdown through a sandboxed parser (no raw-HTML pass-through).

Article markdown can embed charts: fence a block as chart with a Vega-Lite JSON spec and kiri renders it inline as a chart — bar, line, area, scatter, pie, heatmap, and more. The data must be inline in the spec; a chart that fetches a remote URL is rejected, and an invalid spec degrades to an inline notice without breaking the article.

A failed publish step does not affect the run's status; it's captured for debugging on the run page and filtered from the main step list.

Recommendations — proposed follow-ups

A main step can propose follow-up workflow invocations attached to the run that produced them. A run that aggregates open PRs, for example, can recommend one pr-review invocation per PR with pr_number pre-filled — turning the aggregator's output into a launch pad for one-click follow-ups. Recommendations surface on the run detail page under a Recommended section, each as a trigger button that opens the standard invoke modal with the proposed inputs pre-filled.

To emit recommendations, write JSON Lines to the path in $KIRI_RECOMMENDATIONS_FILE — one object per line:

name: open-prs
steps:
  - sh: |
      gh pr list --json number,title --jq '.[]' | while read -r pr; do
        number=$(echo "$pr" | jq -r .number)
        title=$(echo "$pr" | jq -r .title)
        jq -n --arg n "$number" --arg t "$title" \
          '{title: ("Review PR #" + $n + ": " + $t), workflow: "pr-review", inputs: {pr_number: $n}}' \
          >> "$KIRI_RECOMMENDATIONS_FILE"
      done

Each line has title (required), workflow (required — the workflow name to invoke), description (optional, shown under the title), and inputs (optional { string: string } map pre-filled into the invoke modal). Malformed JSON or schema-failing lines are skipped with a warning; the rest of the file still ingests.

KIRI_RECOMMENDATIONS_FILE is set on main steps: only — not on publish: or summarize:. A failed or cancelled step's file is discarded, even if it wrote lines before failing.

Acting on a recommendation spawns the proposed workflow and pins the spawned run's id to the recommendation row. If the spawned run is later deleted the recommendation becomes triggerable again; if the named workflow is removed from your repo the trigger button disables with a "workflow not found" tooltip.

Trust model

Kiri runs scripts with your user's permissions. scripts/<name>/run.sh and inline sh: steps are shell scripts you wrote into your own repo — kiri does not sandbox them. Treat them like any shell script you'd run yourself: read it before you use it.

The defences kiri does provide are external: the HTTP API binds to 127.0.0.1 only and requires a custom X-Kiri-Client header on state-changing requests, so other browser tabs and arbitrary LAN clients can't trigger workflow runs.
