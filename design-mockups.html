§ 01 · The critique

A workday console with editorial polish.

Kiri isn’t a periodical — it’s how you run and read your workday. The editorial palette and serif / mono pairing should stay; the chrome around them needs to feel like a calm operator’s surface, not a magazine. What follows critiques the current SPA against that posture and against the data Kiri actually stores.

Keep

The editorial palette

Cream-on-black with the warm gold is uncommon and instantly identifiable. It reads as a thoughtful tool, not a SaaS dashboard.

Keep

Fraunces + JetBrains Mono

The serif/mono pairing is doing real work — Fraunces’ optical sizing and italic give you a wide register to play with. You’re not using its full range yet.

Rework

The feed is a monotone list

“Dev / Patch · Dev / Patch · Dev / Patch …” five times in a row, all in the same display type at the same size. The eye finds nothing to land on. Consecutive runs of the same workflow should collapse into one entry whose children are the actual run summaries.

Rework

No sense of when

Run rows mix “1 hour ago” with “yesterday” with “2 days ago” in one continuous strip. Date dividers (driven by startedAt) give the feed an at-a-glance structure without inventing anything.

Lift

The status strip is your motif. Use it more.

That 2px coloured strip is the one bit of chrome unique to Kiri. Wear it at full volume on the run hero (4px), as dots in the sidebar, on workflow run cards, and on the activity table. Make it the through-line.

Lift

Use the gold more bravely

#d6b85a barely appears outside hover states. Promote it to: drop caps on article first paragraphs, section eyebrows, the “Published” affordance on run pages, active sidebar accents. Stop hiding the only colour you’ve committed to.

Rework

Workflow page leads with the code

Opening a workflow drops you onto a 200-line YAML dump. The thing you almost always want is “what did it last do, and is it healthy?” Promote recent runs and a duration sparkline above the definition.

Rework

Articles render as sparse markdown

Digests and patch reports are just markdown — that’s deliberately the only contract. So the job is to set that markdown beautifully: a real H1 lockup, drop cap on the first paragraph, generous line lengths, an auto-derived H2 table of contents in the margin, and stronger list rhythm. Do not invent structured fields the data doesn’t carry.

Rework

Run header buries its outcome

The hero repeats the workflow name (which you just clicked on). The run.summary markdown is what happened — promote it from a section in the body to the hero line, set in editorial type. Workflow name demotes to a breadcrumb.

Lift

The right rail is duplicative

“Recently Published” just mirrors the feed’s articles. Replace with something the feed can’t show with first-class data: pending recommendations across all runs, in-flight runs, and today’s failed runs needing attention.

Lift

The wordmark is a whisper

“kiri” sits alone in the corner. Add a small live status line beneath it (in-flight count, today’s run total, the version), and keep the wordmark tall — it’s the only identity moment in the chrome.

Lift

The sidebar workflows are a flat list

Nine entries with / separators. Lean into the natural grouping (Dev, News, BPP) and add a tiny status dot + last-run mono tail beside each name. Same information, ten times the scannability.

Self-audit — what this document's v1 mocks invented.

A first pass leaned on a “personal newspaper” framing and drew structured fields Kiri doesn’t store. The inventions are listed below so future iterations don’t repeat them. Articles are only published markdown; that contract is load-bearing and isn’t changing.

Cut

The periodical metaphor

Vol. 1 · No. 147 wordmark meta, “Today.” display masthead, “edition closed · 21:00” — Kiri isn’t a daily issue. The wordmark gets a quiet live-status line instead.

Cut

Verdict stamps on PR-review rows

“Approve with nits” only exists if the run’s summary mentions it. There’s no verdict field. The PR card embed (#1045 · join-flex · …) likewise treated an external PR as a Kiri entity. Both go.

Cut

Stat strips on articles & runs

“2 advisories patched · 0 skipped · 78 tests passing · 1 PR opened” rendered as a 4-cell stat lockup. None of those are fields; they only appear in the summarizer’s prose. The same goes for the 84-pt “0 / 1” on Open-PR runs.

Cut

CVE / severity tags & story cards

Patch articles were drawn as “High · CVE-2025-37132” cards. Digest articles were drawn as per-story cards with a pull quote. Articles are markdown; render the markdown well, don’t parse it into pretend structures.

Cut

Invented run-detail verdict hero

“Patched 2 advisories, opened PR #79.” was a sentence I typed. The real run.summary is “autoid-verify-service patched 2 advisories with 0 skipped at …” — that markdown should render in the hero in editorial type.

Cut

Workflow “this week” faux stats

“Advisories patched · 31” / “PRs opened · 15” aren’t fields. The week panel can show only what runs carry: count, status breakdown, average duration, and the duration sparkline.

§ 02 · Mockup

The daily edition.

A periodical you scan, not a log you scroll. Date dividers, a day-brief, differentiated treatments per workflow type, and identical-workflow runs collapsed into one entry.

(6) Every feed row is now kicker · workflow-name · run.summary · article links. No verdict stamps, no PR cards, no source-rail tags — the row is the summary markdown, set well. (7) Right rail trades the duplicative “Recently Published” list for three operator surfaces: pending recommendations across all runs (real), currently-running runs (real), and a needs-attention summary keyed on non-ok runs (real).
§ 03 · Mockup

A real reading view.

The published surface is where Kiri earns the “personal periodical” metaphor. Treat it like one: drop cap, deck, byline, marginalia table of contents, embedded story cards, a pull quote, and a source rail at the foot.
